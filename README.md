# Internal Agency & Partner Task Management Platform

Production-ready internal work management platform for administrators, employees, partner agencies, and service providers.

## Stack

- Frontend: Next.js 15, TypeScript, TailwindCSS, shadcn/ui-style components, React Query, Axios
- Backend: FastAPI, PostgreSQL, SQLAlchemy, Alembic, JWT authentication
- Storage: MinIO-compatible object storage
- Realtime: FastAPI WebSockets
- Deployment: Docker and Docker Compose

## Features

- JWT login, logout, protected routes, refresh token rotation, and password hashing
- Role-based access control for admins, employees, and partner agency users
- Agency isolation for partner users and assigned-task isolation for employees
- Agency, user, and task management
- Task statuses: Pending, Working, Need Information, Completed, Cancelled
- Automatic activity logs for task creation, updates, status changes, messages, and files
- Realtime task chat over WebSockets
- Secure file uploads to MinIO with short-lived presigned download URLs
- In-app notifications for assigned tasks, messages, status changes, and uploaded files
- Responsive light/dark UI for desktop, tablet, and mobile
- OpenAPI docs generated by FastAPI

## Quick Start

```bash
cd agency-task-platform
cp .env.example .env
docker compose up --build
```

Open:

- Web app: http://localhost:3000
- API docs: http://localhost:8000/api/v1/docs
- MinIO console: http://localhost:9001

Default seeded admin:

- Email: `admin@example.com`
- Password: `ChangeMe123!`

Change these values in `.env` before production use.

## Local Development

### Without Docker

Use this mode if Docker Desktop is unavailable or stuck.

Backend:

```bash
cd backend
DATABASE_URL=sqlite:///./local-dev.db STORAGE_BACKEND=local BACKEND_PUBLIC_URL=http://localhost:8000 CORS_ORIGINS=http://localhost:3000,http://127.0.0.1:3000 python -m app.local_dev
DATABASE_URL=sqlite:///./local-dev.db STORAGE_BACKEND=local BACKEND_PUBLIC_URL=http://localhost:8000 CORS_ORIGINS=http://localhost:3000,http://127.0.0.1:3000 uvicorn app.main:app --reload
```

Frontend, in another terminal:

```bash
cd frontend
NEXT_PUBLIC_API_URL=http://localhost:8000/api/v1 npm run dev
```

Open http://localhost:3000 and log in with `admin@example.com` / `ChangeMe123!`.

## Environment

Important production values:

- `SECRET_KEY`: set to a long random value
- `DATABASE_URL`: PostgreSQL SQLAlchemy URL
- `CORS_ORIGINS`: comma-separated allowed frontend origins
- `MINIO_*`: object storage endpoint and credentials
- `ADMIN_EMAIL` / `ADMIN_PASSWORD`: initial administrator account
- `NEXT_PUBLIC_API_URL`: public API URL used by the browser

## Free Online Deployment

Recommended free stack:

- Frontend: Vercel
- Backend: Render free web service
- Database: Neon free PostgreSQL
- File storage: Cloudflare R2 free tier

### 1. Push to GitHub

Create a GitHub repository and push the `agency-task-platform` folder.

### 2. Create Neon Database

Create a free Neon project, then copy the pooled or direct PostgreSQL connection string.

Use it as:

```text
DATABASE_URL=postgresql://...
```

The backend accepts both `postgres://` and `postgresql://` URLs and normalizes them for SQLAlchemy.

### 3. Create Cloudflare R2 Bucket

Create an R2 bucket, then create R2 API credentials.

Use these backend environment variables:

```text
STORAGE_BACKEND=s3
STORAGE_AUTO_CREATE_BUCKET=false
S3_ENDPOINT_URL=https://your-account-id.r2.cloudflarestorage.com
S3_ACCESS_KEY_ID=your-r2-access-key-id
S3_SECRET_ACCESS_KEY=your-r2-secret-access-key
S3_BUCKET=task-attachments
S3_REGION=auto
```

### 4. Deploy Backend on Render

Render can read `render.yaml` from the repository.

Set these Render environment variables:

```text
DATABASE_URL=your-neon-postgres-url
SECRET_KEY=long-random-secret
CORS_ORIGINS=https://your-vercel-app.vercel.app
STORAGE_BACKEND=s3
STORAGE_AUTO_CREATE_BUCKET=false
S3_ENDPOINT_URL=https://your-account-id.r2.cloudflarestorage.com
S3_ACCESS_KEY_ID=your-r2-access-key-id
S3_SECRET_ACCESS_KEY=your-r2-secret-access-key
S3_BUCKET=task-attachments
S3_REGION=auto
ADMIN_EMAIL=your-admin-email
ADMIN_PASSWORD=your-admin-password
```

Render start command:

```bash
sh start.sh
```

The start script runs migrations, seeds the admin, and starts FastAPI.

### 5. Deploy Frontend on Vercel

Create a Vercel project with:

```text
Root Directory: frontend
Framework: Next.js
Install Command: npm ci
Build Command: npm run build
```

Set:

```text
NEXT_PUBLIC_API_URL=https://your-render-api.onrender.com/api/v1
```

After Vercel gives you the frontend URL, add it to the backend `CORS_ORIGINS` value on Render and redeploy/restart the backend.

### 6. Send Agencies the Link

Send agencies the Vercel app URL, not your local `127.0.0.1` URL.

Example:

```text
Link: https://your-vercel-app.vercel.app
Email: agency@example.com
Password: TemporaryPassword123!
```

## API Overview

All API routes are under `/api/v1`.

- `POST /auth/login`
- `POST /auth/refresh`
- `POST /auth/logout`
- `GET /auth/me`
- `GET/POST/PATCH/DELETE /agencies`
- `GET/POST/PATCH/DELETE /users`
- `GET/POST/PATCH/DELETE /tasks`
- `PATCH /tasks/{task_id}/status`
- `GET/POST /tasks/{task_id}/messages`
- `WS /tasks/{task_id}/ws`
- `GET/POST/DELETE /tasks/{task_id}/attachments`
- `GET/PATCH/POST /notifications`
- `GET /dashboard`

The full schema is available at `/api/v1/docs`.

## Security Model

- Admins can view and mutate all records.
- Employees can only view tasks directly assigned to them.
- Partner users can only view tasks assigned to their agency.
- Task-level access is enforced server-side for reads, status updates, messages, files, and WebSockets.
- Uploads are extension allow-listed and size-limited.
- Download links are generated as short-lived MinIO presigned URLs.

## Database

- `users`
- `agencies`
- `tasks`
- `messages`
- `attachments`
- `activity_logs`
- `notifications`
- `refresh_tokens`

## Production Notes

- Put the app behind HTTPS and a reverse proxy.
- Use strong database and MinIO credentials.
- Replace the default admin password immediately.
- Store `.env` in your deployment secret manager.
- Configure backups for PostgreSQL and MinIO volumes.
- Consider external object storage by pointing the `MINIO_*` variables at any S3-compatible provider.
